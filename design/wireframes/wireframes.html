Offline state

OFFLINE

Login requires network. Show error message if offline — do not show spinner indefinitely.

Decisions

DECIDED
  • Auth tokens in-memory (Zustand), refresh token in httpOnly cookie
  • Session persists across home-screen launches via cookie
  • No "remember me" toggle needed — refresh token handles this

Trigger

Decisions

DECIDED

  • Trainer mode 14-day free trial before billing gate
  • This choice should feel weighty — stacked full-height cards, not a tab or dropdown

Offline state

Layout principle

DECIDED

Dashboard is an action hub. Actions lead. KPIs follow.

  • Header strip (compact chips): streak, last session, recent PR — glanceable, not the hero
  • TRAIN — primary action card, always first, command-blue fill
  • PLAN — schedule a future session
  • GOALS — view / update active goal
  • HISTORY — jump to last session
  • BUILD — build a template
  • Log Snapshot — secondary, bottom
  • KPI widgets — user-configured, below the fold

TRAIN card states

  • Default: "Start Training" — no planned session
  • Planned: "Begin: Push Day A ×" — a planned session exists
  • In progress: "Resume: Push Day A ×" — session already started

× dismisses the planned/in-progress session after confirmation.

KPI pinning

Configured from Goals screen, not from Dashboard. Dashboard itself is configuration-free. Pinned KPI chips appear in the header strip alongside streak + last session.

Offline state

OFFLINE

All action cards work from cache. TRAIN works offline (session queued locally). PLAN/GOALS/BUILD require network for mutations. KPI strip shows stale timestamp.

Install prompt

INSTALL

Does NOT fire on dashboard. Fires after first completed session on Session Summary.

Layout principle

DECIDED

Same action-hub structure as Athlete. Trainer-specific actions replace athlete ones. KPIs are below the fold and user-configured.

  • Header strip: active client count, at-risk count, my last session
  • At-risk alert: Ember Red, prominent — but not the primary CTA
  • TRAIN — My Training (isSelf session)
  • PLAN — plan a client session
  • CLIENTS — roster with at-risk summary
  • HISTORY — most recent client session
  • BUILD — build a template
  • KPI widgets — configurable, below fold

At-risk alert

  • Threshold: 21+ days without a session
  • Dismissal: per-session only — resets on next app open (by design)
  • Also appears as chip in header strip for quick reference

Decisions

Reports: email-only (Resend). No in-app report preview in initial scope.

Workout type

DECIDED

No upfront type selection. workoutType is inferred from the exercise record when each exercise is added. A mixed session (e.g. resistance + stretching) carries the type per exercise, not per session.

Two paths

  • From template: picks exercises + targets from the template record, workoutType carried from template_exercise
  • Blank: opens live session with empty exercise list — user adds exercises from the library

Trigger

Opened via bottom-sheet from any "Start Session", TRAIN card, or "New" button. Also reachable at /session/new.

Offline state

OFFLINE

Fully works offline. Template list from cache. Session created in local queue, syncs on reconnect.

Key UX decisions

  • "Last time" context must be prominent — currently a small text line, needs more visual weight (see Hevy comparison)
  • Past sets: color-coded (emerald = hit/surpass, amber = missed) — TrainerApp unique
  • Future sets: greyed placeholders showing upcoming targets
  • Active set: full hero with large inputs, pre-filled from target or last set

Offline state

Timing

  • Requires: supporting browser + beforeinstallprompt event captured
  • Does not fire on first load
  • Non-recurring: dismissed = never shown again
  • iOS: no native prompt — show manual instructions sheet instead (Phase 12)

Copy

"Train anywhere, even without signal. Your session data never disappears."

CTA: "Add to Home Screen" (not generic "Install")

Dismiss: "Not now" — permanent

Four states

  • Offline: "Offline — changes will sync when you reconnect" + queued count badge
  • Syncing: "Syncing…" with spinner
  • Queued: "N changes queued"
  • Error: "Sync failed — tap to retry"

Behavior

Banner appears at top of content area (below status bar, above page header). Dismissible? No — stays until reconnected. Auto-dismisses after successful sync.

Core offline guarantee

OFFLINE

Set logging, exercise browsing, template use — all work fully offline. Queue persists to localStorage across app restarts.

Screens that are blocked offline

  • Reports (email send requires network)
  • Login / Onboarding
  • Creating new exercises (backend enrichment required)
  • Adding new clients

Pattern

Empty state with icon + explanation copy + "available when you reconnect" badge. Never a spinner or an error — the user is informed, not confused.
