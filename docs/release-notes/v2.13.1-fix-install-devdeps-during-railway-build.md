# v2.13.1 тАФ fix: install devDeps during Railway build so tsc is available for shared prepare

> **Type:** Hotfix
> **Date:** 2026-05-04
> **Base:** v2.13.0 (security hardening)

---

## Problem

Railway builds were failing with:

```
devDependencies: skipped because NODE_ENV is set to production

packages/shared prepare$ tsc -p tsconfig.build.json
ELIFECYCLE  Command failed.
```

Railway injects `NODE_ENV=production` into the build environment. pnpm treats this as an implicit `--prod` flag and skips all `devDependencies` across the workspace. `typescript` is a `devDependency` in `packages/shared`, so `tsc` was not on the path when the `prepare` script ran during `pnpm install`.

The `packages/shared` prepare script (`tsc -p tsconfig.build.json`) is required to compile the shared package to CJS before the backend can build тАФ it is the fix for the duplicate Zod instance problem documented in `docs/RAILWAY_ERRORS.md`. Without it, the entire Railway deploy fails.

## Root Cause

`packages/shared/package.json` correctly lists `typescript` as a `devDependency`. This is semantically correct тАФ TypeScript is a build-time tool. However, `NODE_ENV=production` during Railway's build phase caused pnpm to skip it, making `tsc` unavailable to the `prepare` script.

## Fix

**File changed:** `railway.json`

Prefixed the install step with `NODE_ENV=development` so pnpm installs all dependencies (including devDependencies) during the build phase. The `&&`-chained build commands inherit Railway's original `NODE_ENV=production`, so the backend compiles correctly in production mode.

```diff
- "buildCommand": "pnpm install --frozen-lockfile && pnpm --filter @trainer-app/shared build && pnpm --filter backend build"
+ "buildCommand": "NODE_ENV=development pnpm install --frozen-lockfile && pnpm --filter @trainer-app/shared build && pnpm --filter backend build"
```

## Why This Is Safe

- `NODE_ENV=development` applies only to the `pnpm install` process (inline env syntax). It does not propagate to the subsequent commands in the chain.
- The `pnpm --filter @trainer-app/shared build` and `pnpm --filter backend build` steps run with Railway's `NODE_ENV=production` unchanged.
- No lockfile changes required тАФ only the build invocation changes.
- No semantic change to the shared package or its dependency declarations.
