# v2.12.0 Scope — Progress Media + Coach Challenges

> **Status:** Revised draft (v2)
> **Depends on:** Refresh token bug fix (pre-requisite — see § Appendix A)

---

## Design Philosophy

v2.12.0 adds two capabilities that serve the same narrative question: *"Is this client moving forward?"*

1. **Progress media** — visual evidence of change over time (the "then vs now" that numbers alone can't capture)
2. **Coach challenges** — goals with deadlines that give athletes something concrete to chase

Both reinforce the Capture → Compare → Communicate loop. Photos make snapshots richer. Challenges make goals actionable. Neither is a standalone social feature — they feed the existing progress engine. Social sharing consumes this data later (v2.13.0); it doesn't shape how we store it now.

### Privacy-first media

Progress photos are intimate. The default is **private** — photos exist for the trainer and athlete's own use only. A sharing preference controls what's even eligible to leave the app. This isn't a toggle on individual photos after they exist; it's a gate that prevents photos from reaching a shareable state unless the user explicitly opens that gate.

## Feature 1: Progress Photos

### What it is

Photos attached to client snapshots. The snapshot table already captures body comp, circumference, cardiovascular, functional, and subjective data. Progress photos add the visual dimension — front/side/back poses that, compared across months, tell the story numbers sometimes miss.

Photos live on the client profile and within the snapshot record. They're personal records first, potential social content second.

### Who uploads

| Mode | Who takes the photo | Why |
|---|---|---|
| **Trainer** | Trainer uploads for the client | Trainer controls the session, takes consistent poses, ensures quality |
| **Athlete** | Athlete uploads their own | They're self-coaching — no one else is there |

Clients never upload directly. In trainer mode, media capture is the trainer's responsibility. The athlete is their own trainer, so they handle their own uploads.

### Client opt-out

Clients can decline progress photos. A `progressPhotosOptedOut` boolean on the `clients` table (default `false`). When true, the photo capture section doesn't appear in the snapshot form for that client, and the trainer sees a note: "Photos disabled at client's request." The trainer can toggle this in the client's settings — it's a conversation between trainer and client, not a client-facing UI (no client portal yet).

### Photo sharing preferences

A trainer-level preference controls the sharing posture for all progress photos. This is a gate, not a per-photo toggle.

| Setting | Meaning |
|---|---|
| `private` (default) | No photos are shareable. The social share feature (v2.13.0) has nothing to work with. |
| `share_selected` | Only photos explicitly marked as shareable by the trainer/athlete appear in the share pool. The "toShare" pool is a curated subset. |
| `share_all` | All progress photos are available for social sharing. |

New field on `trainers` table: `photoSharingPreference text NOT NULL DEFAULT 'private'`

New shared enum:

```ts
export const PhotoSharingPreferenceEnum = z.enum(['private', 'share_selected', 'share_all'])
```

When `share_selected` is active, each `snapshot_media` record gets a `shareable` boolean (default `false`). The trainer/athlete explicitly marks individual photos as shareable. Only these flagged photos are eligible when the social share feature lands in v2.13.0. This is an extra layer of security — even if the share feature has a bug, un-flagged photos can't leak.

### Why snapshots (not sessions)

Snapshots are the right home because they're the existing "point-in-time measurement" entity. A progress photo is a measurement — it just happens to be visual. Attaching photos to sessions would scatter them across dozens of records and make comparison views painful to build. Snapshots are already taken at phase transitions and monthly intervals, which is the natural cadence for progress photos.

### Schema

New table — `snapshot_media`:

```
snapshot_media
├── id                  uuid PK
├── snapshotId          uuid FK → client_snapshots.id (CASCADE)
├── pose                snapshot_pose NOT NULL — 'front' | 'side_left' | 'side_right' | 'back' | 'custom'
├── cloudinaryUrl       text NOT NULL
├── cloudinaryPublicId  text NOT NULL
├── width               integer
├── height              integer
├── caption             text — optional note ("Week 4, post-cut")
├── shareable           boolean NOT NULL DEFAULT false — only relevant when sharing pref = 'share_selected'
├── orderIndex          integer NOT NULL DEFAULT 0
├── createdAt           timestamp NOT NULL DEFAULT now()
```

New column on `clients` table:

```
progressPhotosOptedOut  boolean NOT NULL DEFAULT false
```

New column on `trainers` table:

```
photoSharingPreference  text NOT NULL DEFAULT 'private'
```

New shared enums:

```ts
export const SnapshotPoseEnum = z.enum(['front', 'side_left', 'side_right', 'back', 'custom'])
export const PhotoSharingPreferenceEnum = z.enum(['private', 'share_selected', 'share_all'])
```

### Cloudinary folder structure

```
trainer-app/
├── exercises/<exerciseId>/                          ← existing
└── clients/<clientId>/snapshots/<snapshotId>/       ← new
```

### API routes

| Method | Path | Purpose |
|---|---|---|
| POST | `/snapshots/:id/media` | Upload photo(s) to a snapshot (multipart) |
| DELETE | `/snapshot-media/:id` | Remove a single photo |
| PATCH | `/snapshot-media/:id` | Update caption, shareable flag |
| GET | `/clients/:id/progress-photos` | All photos for a client, grouped by snapshot date |

The upload route reuses `uploadBuffer()` and `validateMediaFile()` from `cloudinary.service.ts` with a different folder path. The progress-photos GET joins `snapshot_media` → `client_snapshots` and returns photos grouped by `capturedAt` date, powering the comparison view without loading full body comp data. Response includes `shareable` status so the UI can show the share indicator when `share_selected` is active.

### Frontend

**Snapshot capture flow (enhanced):**
The existing snapshot form gets a "Photos" section at the top. If the client has `progressPhotosOptedOut = true`, the section is replaced with a disabled note. Otherwise: camera icon buttons for each pose (front, side L, side R, back). Tapping one opens the device camera (`<input type="file" accept="image/*" capture="environment">`). Photos upload immediately on capture; a thumbnail replaces the camera icon. Tap thumbnail to retake or delete.

For athletes (self-tracking): `capture="user"` to default to front-facing camera, since they're photographing themselves.

When `photoSharingPreference = 'share_selected'`, each uploaded thumbnail gets a small share toggle (eye icon). Tapping it sets `shareable = true` on that photo. Visual distinction: shareable photos get a subtle badge; private photos don't. This is the curation step — the trainer/athlete decides in the moment which photos are share-worthy.

**Progress comparison view:**
`ProgressPhotoTimeline` on the client profile Overview tab. Shows the most recent photo set, with a "Compare" button opening a side-by-side slider (before/after with a draggable divider). Date pickers select which two snapshots to compare.

**Preferences section:**
New "Privacy" subsection in `/preferences` — "Progress Photo Sharing" with three radio options: Private (default), Share Selected, Share All. Explanation text under each option so the trainer understands the implication before choosing.

**Components:**
- `SnapshotPhotoCapture` — camera buttons per pose, upload handling, thumbnail previews, share toggle
- `ProgressPhotoTimeline` — chronological photo grid on client profile
- `PhotoComparisonSlider` — before/after slider with date selectors
- `ProgressPhotoModal` — full-screen photo viewer with swipe between poses

### Scoping decisions

- **Images only for progress.** Video is for form checks (Feature 2), not body composition.
- **No AI body analysis.** The comparison slider is the v1 of visual progress. Photos are stored — AI analysis can be layered on later without schema changes.
- **Social share consumes this data in v2.13.0.** We're building the data layer and privacy controls now. The share UI (generate card image, Web Share API) is next version.

## Feature 2: Session Form Check Clips

### What it is

Short video clips (≤30 seconds) attached to a session exercise. The trainer records a client's form during a set, reviews it between sessions, or references it in the monthly report. This is a form check tool, not a content creation platform.

### Inline camera experience

Rather than punting to the native camera app via `<input type="file">`, we use the **MediaStream API** (`navigator.mediaDevices.getUserMedia`) to render a live camera viewfinder inline within the session UI. This keeps the trainer inside the app flow — they don't context-switch to the camera app and back.

**How it works:**

1. Trainer taps the camera icon on an exercise's accordion header
2. A bottom sheet slides up with a live camera preview (rear camera default via `facingMode: 'environment'`)
3. Camera toggle button to flip front/rear
4. For **photo**: single capture button → `canvas.toBlob()` → upload
5. For **video**: record button starts a `MediaRecorder`, timer counts up to 30s, stop button (or auto-stop at 30s) → `Blob` → upload
6. Preview of the captured media before confirming upload
7. Upload runs in background; sheet closes; badge count on the exercise updates

**Key browser API usage:**
- `getUserMedia({ video: { facingMode } })` — camera access
- `MediaRecorder` — video recording to WebM/MP4
- `canvas.drawImage()` + `toBlob()` — still capture from video stream
- Video duration enforced by `setTimeout` at 30s + `MediaRecorder.stop()`

**Fallback:** If `getUserMedia` is unavailable (older browsers, denied permissions), fall back to `<input type="file" accept="image/*,video/*" capture="environment">`. The UI shows the file input instead of the viewfinder. This ensures the feature works everywhere, just with a less seamless experience on older devices.

### Why session exercises (not sets)

A form check video applies to how the client performs the movement in general during that session — not to a specific set. Attaching to `session_exercises` keeps the data model clean and avoids multiplying storage per set.

### Schema

New table — `session_exercise_media`:

```
session_exercise_media
├── id                  uuid PK
├── sessionExerciseId   uuid FK → session_exercises.id (CASCADE)
├── mediaType           media_type NOT NULL — 'image' | 'video' (reuse existing enum)
├── cloudinaryUrl       text NOT NULL
├── cloudinaryPublicId  text NOT NULL
├── durationSeconds     integer — for video, null for image
├── caption             text
├── createdAt           timestamp NOT NULL DEFAULT now()
```

Cloudinary folder:

```
trainer-app/clients/<clientId>/sessions/<sessionId>/<sessionExerciseId>/
```

### API routes

| Method | Path | Purpose |
|---|---|---|
| POST | `/session-exercises/:id/media` | Upload clip/photo during live session |
| DELETE | `/session-exercise-media/:id` | Remove media |
| GET | `/session-exercises/:id/media` | List media for an exercise instance |

### Frontend

**During live session:**
Camera icon on each exercise's accordion header. Tapping opens the inline camera bottom sheet (see above). Badge count shows attached clips. Upload is non-blocking — the trainer can keep logging sets while the upload finishes.

**Post-session / history view:**
`SessionHistoryPanel` shows media thumbnails inline with each exercise's performance data. Tapping opens full-screen playback. Video uses the native `<video>` element — Cloudinary handles transcoding.

**Components:**
- `InlineCameraSheet` — bottom sheet with live viewfinder, photo capture, video recording (30s cap), preview, confirm/retake, camera flip. Reusable — progress photos could adopt it later too.
- `FormCheckBadge` — small indicator on exercise accordion showing media count
- `MediaPlaybackModal` — full-screen image/video viewer

### Scoping decisions

- **30-second cap on video.** Enforced client-side via `MediaRecorder` timer. Keeps storage costs predictable.
- **Inline camera via MediaStream API.** Feels native to the app, avoids context-switching. Falls back to file input gracefully.
- **No video annotation/drawing.** Valuable future premium feature for trainers — circling the knee caving in, drawing the bar path. Separate version, trainer-only tier feature.

## Feature 3: Coach Challenges

### What it is

A challenge is a measurable goal with a deadline. "10 unassisted pull-ups by June 30th." It has a target metric, a deadline, and tracks progress automatically from session data where possible.

Challenges fill the gap between goals (text statements of intent, no finish line) and templates (structured plans, no success criteria). A challenge has both: a concrete target and a date.

### Two interfaces, shared engine

Both trainers and athletes create challenges, but the interface reflects their context:

| | Trainer creating for a client | Athlete creating for themselves |
|---|---|---|
| **Who** | Trainer selects a client, sets the challenge | Athlete sets their own challenge (they're their own coach) |
| **Where** | Client profile → Challenges tab | Dashboard or self-client profile → Challenges section |
| **Tone** | Coach-to-athlete ("I'm setting this for you") | Self-directed ("I'm committing to this") |
| **Backend** | Same `challenges` table, same routes | Same `challenges` table, same routes |

The backend is identical — a challenge is always `clientId` + `trainerId` scoped. For athletes, both point to the self-client. The frontend has separate entry points and slightly different copy, but shares all components and query hooks.

**Shared modules:** `ChallengeProgressCard`, `ChallengeForm`, all query hooks (`useChallenges`, `useCreateChallenge`, `useUpdateChallenge`), the auto-detection logic, the progress bar component.

**Separate interfaces:** The challenge creation entry point (where in the UI the "New Challenge" button lives), the empty state copy, and any contextual help text. The athlete version doesn't reference "clients" — it says "your challenge." The trainer version says "challenge for [client name]."

### Schema

New table — `challenges`:

```
challenges
├── id              uuid PK
├── clientId        uuid FK → clients.id (CASCADE)
├── trainerId       uuid FK → trainers.id (CASCADE)
├── title           text NOT NULL — "10 unassisted pull-ups"
├── description     text — optional context, instructions, motivation
├── metricType      challenge_metric_type NOT NULL
├── exerciseId      uuid FK → exercises.id (SET NULL) — optional, ties to specific exercise
├── targetValue     real NOT NULL — the number to hit
├── targetUnit      text — 'lbs' | 'kg' | 'reps' | 'km' | 'mi' | 'seconds' | 'sessions' | null
├── currentValue    real NOT NULL DEFAULT 0 — latest progress
├── deadline        text NOT NULL — 'YYYY-MM-DD'
├── status          challenge_status NOT NULL DEFAULT 'active'
├── completedAt     timestamp
├── createdAt       timestamp NOT NULL DEFAULT now()
├── updatedAt       timestamp NOT NULL DEFAULT now()
```

New shared enums:

```ts
export const ChallengeMetricTypeEnum = z.enum([
  'weight_lifted',       // heaviest single lift for an exercise
  'reps_achieved',       // max reps in one set for an exercise
  'distance',            // total or single-effort distance
  'duration',            // time-based (plank hold, mile time)
  'sessions_completed',  // N sessions in a period
  'qualitative',         // manual progress — trainer/athlete judgment ("improve squat depth")
])

export const ChallengeStatusEnum = z.enum(['active', 'completed', 'expired', 'cancelled'])
```

Note: `qualitative` replaces the earlier `custom` label. It's clearer about what it is — the trainer/athlete assesses progress by judgment rather than by number. The `targetValue` for qualitative challenges represents a scale (e.g., 1–10 self-assessed), and `currentValue` is updated manually via `PATCH /challenges/:id`.

### Auto-detection logic

When a set is logged (`POST /session-exercises/:id/sets`), the backend checks if the client has any `active` challenges tied to that exercise:

| metricType | Trigger | Update rule |
|---|---|---|
| `weight_lifted` | Set logged for matched exercise | `currentValue = max(currentValue, set.weight)` |
| `reps_achieved` | Set logged for matched exercise | `currentValue = max(currentValue, set.reps)` |
| `distance` | Set logged for matched exercise | `currentValue = max(currentValue, set.distance)` |
| `duration` | Set logged for matched exercise | `currentValue = max(currentValue, set.durationSeconds)` |
| `sessions_completed` | Session completed for client | `currentValue += 1` |
| `qualitative` | Manual update only | Trainer/athlete patches via API |

If `currentValue >= targetValue`, set `status = 'completed'` and `completedAt = now()`.

Expiry check runs daily via the existing BullMQ scheduler (alongside at-risk alerts): if `deadline < today` and `status = 'active'`, flip to `expired`.

### API routes

| Method | Path | Purpose |
|---|---|---|
| GET | `/clients/:id/challenges` | List challenges for a client (filterable by status) |
| POST | `/clients/:id/challenges` | Create a challenge |
| PATCH | `/challenges/:id` | Update challenge (progress, deadline, status) |
| DELETE | `/challenges/:id` | Soft-cancel (set status to cancelled) |

These routes work identically for trainer-managed clients and self-clients. The `authenticate` + ownership check middleware handles scoping. No separate athlete routes needed.

### Frontend

**Trainer flow:**
"New Challenge" button on client profile → Challenges tab → `ChallengeForm` bottom sheet. Title, metric type picker (drives which subsequent fields appear), exercise selector (filtered by workout type, hidden for `sessions_completed` and `qualitative`), target value + unit, deadline date picker.

**Athlete flow:**
"New Challenge" button on the dashboard (or self-client profile). Same `ChallengeForm` bottom sheet, same fields. Copy says "Set a challenge" instead of "Challenge for [name]". The self-client ID is automatically used — no client selector.

**Challenge progress cards:**
Shared `ChallengeProgressCard` component. Progress bar fills proportionally, color shifts toward green near completion. If deadline is within 7 days and progress <50%, subtle urgency indicator (same pattern as at-risk alerts). Tappable for detail + edit.

**Client profile — Challenges tab:**
New tab on the client profile panel (alongside Overview, Timeline, Baseline). Active challenges as cards, completed/expired collapsed below.

**Athlete dashboard — Challenges section:**
Active challenges rendered as a widget-like section on the dashboard. Same `ChallengeProgressCard` component, different container.

**Components (shared):**
- `ChallengeForm` — bottom sheet, metric-type-aware fields, exercise picker, deadline
- `ChallengeProgressCard` — progress bar, days remaining, status badge
- `ChallengeDetailPanel` — full detail view with edit capabilities and progress history

**Query hooks (shared):**
- `useChallenges(clientId)` — list with status filter
- `useCreateChallenge()` — mutation
- `useUpdateChallenge()` — mutation (progress, status, fields)

### Scoping decisions

- **No gamification beyond the progress bar.** Badges, XP, streaks are v2.13.0. Challenges are a utility feature.
- **No client notifications.** Monthly report includes challenge summary — that's the communication channel for now. Future options: client portal, SMS push, or other lightweight touchpoints. The shape of client communication is still open.
- **Qualitative type for manual/judgment-based progress.** Not every challenge maps to a clean metric. "Improve squat depth" is the trainer's or athlete's call. `qualitative` makes this explicit rather than hiding it behind a generic "custom" label.
- **No challenge templates.** Premature. Build the primitive, observe what trainers and athletes actually create, then template the patterns that emerge.

## Appendix A: Refresh Token Bug — Debugging Steps

This must be resolved before v2.12.0 ships. New features can be developed in parallel, but the app can't go to wider beta with broken session persistence.

### Step 1 — Verify cookie storage

1. Open the production app in Chrome
2. Log in successfully
3. **Before refreshing**, open DevTools → Application → Cookies
4. Look for the Railway backend domain (e.g., `your-app.up.railway.app`)
5. Check if `refresh_token` cookie exists

**If the cookie is NOT there:** The browser is rejecting the cross-origin cookie. Modern browsers (Chrome 115+, Safari 17+) block third-party cookies by default even with `SameSite=None; Secure`. This is the most likely cause.

**If the cookie IS there:** The issue is in `attemptTokenRefresh()` — either it's not sending `credentials: 'include'`, or the backend's `/auth/refresh` route isn't reading the cookie correctly. Check the Network tab for the refresh request and its response.

### Step 2 — If cookie is rejected (likely path)

**Option A — Proxy approach (recommended):**
Add a Vercel rewrite that proxies `/api/*` requests to the Railway backend. This makes the cookie first-party (same domain as the frontend). The cookie `path` changes from `/api/v1/auth` to the proxied path, and `sameSite` can be `lax` instead of `none`.

```json
// apps/frontend/vercel.json
{
  "rewrites": [
    { "source": "/api/:path*", "destination": "https://your-app.up.railway.app/api/:path*" }
  ]
}
```

Then update `refreshTokenCookieOptions()` in `auth.service.ts`:
- `sameSite: 'lax'` (not `none`)
- `secure: true`
- `domain`: remove (defaults to the response origin, which is now the Vercel domain)

Update `CORS_ORIGIN` on Railway since requests now come from Vercel's edge, not the browser directly.

**Option B — Token in response body:**
Return the refresh token in the login/refresh response body instead of a cookie. Store it in memory (Zustand, alongside the access token). Send it as a header or body parameter on refresh requests. This avoids cookies entirely but means refresh tokens don't survive page reload — same as current breakage, so only useful if combined with a short-lived localStorage fallback (with the security tradeoff that implies).

**Recommendation:** Option A (proxy) is the clean fix. It's a Vercel config change + one backend function update. No security model changes.

### Step 3 — Verify the fix

After applying either option:
1. Log in
2. Confirm cookie is present (if Option A) or token is in memory (if Option B)
3. Hard refresh (Cmd+R) — should stay logged in
4. Close and reopen browser — should stay logged in (within 7-day TTL)
5. Wait 15+ minutes — access token expires, silent refresh should issue a new one without redirect to login
